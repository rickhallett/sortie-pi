### Findings

#### [ISSUE-001] Architecture / Security: Unenforced Domain Lock (Critical)
- **Location**: `src/validation/pipeline.ts:251`
- **Confidence**: High
- **Issue**: The `createDomainLock` function is a factory that returns a tool checker function. However, the pipeline calls `createDomainLock` but never captures its return value or passes it to the lead session or custom tools. The lock is effectively dead code, meaning the lead session is still operating with unrestricted tools (leaving VULN-003 unresolved).
- **Impact**: The lead agent can bypass the sandbox entirely, executing write operations outside the specified domain lock.
- **Evidence**:
  ```typescript
  // Create domain lock scoped to this run's directory (VULN-003)
  createDomainLock(
    [`${config.deposition_dir}/${runIdStr}/**`],
    cwd,
  );

  // Create and invoke lead session
  const { session: leadSession, dispose: disposeLead } = await createLeadSession(
    config.debrief,
    { cwd, customTools: sortieCustomTools }, // No lock is applied here!
  );
  ```
- **Fix**: Update `createLeadSession` or the tool configuration to accept and enforce the domain lock function returned by `createDomainLock`.

#### [ISSUE-002] Security: Command Injection via git diff (High)
- **Location**: `src/validation/pipeline.ts:62`
- **Confidence**: High
- **Issue**: `computeDiff` interpolates the `branch` argument directly into a shell command using `execSync`. The input string is not sanitized.
- **Impact**: If `branch` contains shell metacharacters (e.g., `feature; rm -rf /`), arbitrary commands will be executed on the host system running the pipeline.
- **Evidence**:
  ```typescript
  return execSync(`git diff --merge-base origin/main ${branch}`, {
    cwd,
    encoding: "utf-8",
    maxBuffer: 10 * 1024 * 1024,
  }).trim();
  ```
- **Fix**: Use `execFileSync` or `spawnSync` from `node:child_process` and pass the arguments as an array instead of a single string evaluated by the shell.

#### [ISSUE-003] Bugs / Correctness: Memory Leak on Lead Session Error (High)
- **Location**: `src/validation/pipeline.ts:265`
- **Confidence**: High
- **Issue**: The lead session disposal function (`disposeLead()`) is invoked after `await leadSession.prompt(debriefPrompt)`. If the prompt times out or throws an error (e.g., API failure), execution jumps straight to the `catch` block. The disposal function is never called.
- **Impact**: LLM sessions and any associated resources (e.g., background watchers, open handles) within the Pi SDK are leaked. Over time or across multiple failures, this can exhaust memory and resources.
- **Evidence**:
  ```typescript
  try {
    // ...
    await leadSession.prompt(debriefPrompt);
    const debriefResponse = leadSession.getLastAssistantText();
    disposeLead(); // Skipped if prompt() throws!
  } catch (err) {
    // ...
  }
  ```
- **Fix**: Move `disposeLead()` into a `finally` block to ensure deterministic teardown regardless of network or API errors.

#### [ISSUE-004] Architecture / Correctness: Lead Session Metrics Lost (Medium)
- **Location**: `src/validation/pipeline.ts:316` and `349`
- **Confidence**: High
- **Issue**: The lead (debrief) session's wall time and token costs are not captured. They are hardcoded to `0` in the debrief attestation and missing entirely from the ledger entry's tokens summary (which only aggregates the reviewer outputs).
- **Impact**: Inaccurate telemetry. The overall cost and duration of the validation pipeline are under-reported, obscuring the actual operational expense.
- **Evidence**:
  ```typescript
  // Pipeline Step 8
  writeAttestation(runDirPath, {
    step: "debrief",
    verdict: verdict.verdict,
    findings_count: verdict.findings.length,
    tokens: 0,        // Hardcoded
    wall_time_ms: 0,  // Hardcoded
    // ...
  });
  ```
- **Fix**: Before disposing the lead session, call `getSessionStats()` to capture token usage and cost. Time the execution of the debrief prompt to capture the `wall_time_ms`. Inject these metrics into the final verdict attestation and append them to the ledger payload.

#### [ISSUE-005] Testability: Flawed Domain Lock Test (Medium)
- **Location**: `src/validation/pipeline.test.ts:517`
- **Confidence**: High
- **Issue**: The test `"creates a run-scoped domain lock before creating the lead session"` verifies only that the `createDomainLock` factory function is *called* (via a spy/mock). It does not test that the tools supplied to `createLeadSession` actually use this lock.
- **Impact**: This false positive allowed a severe security flaw (the completely unapplied domain lock in ISSUE-001) to slip into the `main` branch unnoticed.
- **Evidence**:
  ```typescript
  expect(harness.domainLockCalls).toEqual([
    {
      patterns: [`.sortie/${workspace.runId}/**`],
      cwd: workspace.root,
    },
  ]);
  // No assertion that leadSession uses the lock
  ```
- **Fix**: Update the test harness to inspect the tools passed to `createLeadSession` and verify that `beforeToolCall` or an equivalent interceptor maps to the domain lock logic.

#### [VERIFY-001] Synchronous File Writes in Loop
- **Location**: `src/validation/pipeline.ts:205`
- **Question**: `writeFileSync` and `writeAttestation` (which internally uses `writeFileSync`) are called within a synchronous loop over `rosterEntries`. While fine for typical pipelines with 2-4 reviewers, will this cause event loop blocking that impacts performance in environments running with large rosters or high concurrency? Should these be converted to asynchronous `writeFile` operations with `Promise.all`?
